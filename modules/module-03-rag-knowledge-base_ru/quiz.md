# Модуль 3: Квиз — RAG и база знаний

### В1: Что делает RAG?
Без RAG LLM ответит из своей «памяти» → вероятная **галлюцинация**. С RAG система сначала находит реальные документы, затем LLM отвечает на их основе.

### В2: Порядок ingestion-пайплайна?
**Ответ:** Извлечение текста → Чанкинг → Эмбеддинг → Сохранение в Weaviate

### В3: Чанк содержит «...16 недель оплачиваемого отпуска. Это применяется» — что не так?
**Ответ:** Чанк обрезан посередине предложения → потеря контекста. Поиск по «сколько недель отпуска» может не найти этот чанк.

### В4: Что такое Golden Set?
**Ответ:** Набор пар вопрос-документ с заведомо правильными ответами. Используется для регрессионного тестирования — если пара ломается, значит что-то изменилось в RAG-пайплайне.

### В5: Пользователь спрашивает о «декретном отпуске», но KB возвращает документы об «ежегодном отпуске». Причина?
**Ответ:** Проблема качества эмбеддингов — «декретный» и «ежегодный» слишком близки в векторном пространстве.

### В6: Загрузили V2 документа, но поиск показывает V1. Где проверять?
**Ответ:** Ingestion-пайплайн: завершилась ли задача? Заменены ли старые чанки?

### В7: Почему OCR-тестирование критично для приложения Clerk (Релиз 3)?
**Ответ:** Ошибки OCR (неверно прочитанные цифры, текст) приводят к неверным ответам по финансовым/юридическим документам.
