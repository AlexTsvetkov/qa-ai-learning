# Модуль 1: Теория — Архитектура платформы и основы AI

## 1.1 Платформа: обзор

Вы тестируете **корпоративную AI-платформу**, развёрнутую on-premises (без облака, без внешних API). Три приложения:

| Приложение | Релиз | Назначение | Источники данных |
|-----------|-------|-----------|-----------------|
| **AI HR Ассистент** | Релиз 1 | Ответы по HR-политикам | HR-документы, справочники |
| **AI Финансовый Ассистент** | Релиз 2 | Финансовая отчётность, OLAP | Финансовые данные, MS SQL |
| **AI Клерк** | Релиз 3 | Обработка документов, OCR | Сканированные документы |

**Ключевое ограничение:** Всё работает on-premises. Данные не покидают сеть компании. Используются **open-source модели** (Llama 3) на собственном оборудовании.

## 1.2 Сквозной поток данных

```
User → React FE → .NET Backend → Agent Runtime (MCP Host)
                                        │
                              ┌─────────┼─────────┐
                              ▼         ▼         ▼
                         MCP Client  MCP Client  MCP Client
                              │         │         │
                              ▼         ▼         ▼
                         MCP Server  MCP Server  MCP Server
                         (Retrieval) (Query Gen) (External)
                              │         │
                              ▼         ▼
                     Knowledge Base   LLM Gateway
                     (Weaviate)       (vLLM/Llama 3)
```

### Шаг за шагом:

1. **Пользователь** вводит вопрос в браузере
2. **React Frontend** отправляет запрос в .NET Backend API
3. **.NET Backend** аутентифицирует пользователя (Active Directory), проверяет права
4. **Agent Runtime** (MCP Host) решает, какие инструменты вызвать
   - **Prompt Resolver** — выбирает системный промпт (HR/Finance/Clerk)
   - **Trace Collection** — логирует каждый шаг в Langfuse
   - **MCP Clients** — интерфейсы для вызова MCP-серверов
5. **MCP Client** вызывает нужный **MCP Server**
6. **Knowledge Base** выполняет поиск: эмбеддинг вопроса → поиск в Weaviate → RBAC/ABAC фильтры
7. **LLM Gateway** отправляет вопрос + найденный контекст в **vLLM** (Llama 3)
8. Ответ возвращается: LLM → Agent → Backend → Frontend → Пользователь

> 🎯 **Ментальная модель QA:** Каждый баг живёт где-то в этой цепочке. Ваша задача — определить, **где** произошёл сбой.

## 1.3 Компоненты (взгляд QA)

### Frontend (React)
**Что тестируем:** чат-интерфейс, стриминг ответов, загрузка файлов, история
**Типичные баги:** 🐛 стриминг обрывается, длинные ответы переполняют контейнер

### Backend (.NET/C#)
**Что тестируем:** API эндпоинты, аутентификация AD, таймауты, обработка ошибок
**Типичные баги:** 🐛 500 вместо описательной ошибки, протухание токена

### Agent Runtime (MCP Host)
**Что тестируем:** выбор правильного инструмента, обработка отказов, цепочки вызовов, границы промпта
**Типичные баги:** 🐛 агент вызывает не тот инструмент, бесконечные циклы, утечка за рамки промпта

### MCP (Model Context Protocol)
**Что тестируем:** обнаружение инструментов, вызов с параметрами, обработка ошибок, таймауты
**Типичные баги:** 🐛 MCP-сервер возвращает битый JSON, инструменты не найдены

### LLM и модели
| Компонент | Назначение | Эндпоинт |
|-----------|-----------|----------|
| LLM Gateway | Прокси для маршрутизации | Внутренний |
| vLLM | Llama 3 — генерация текста | `/chat` |
| HuggingFace TEI | Эмбеддинги | `/embed` |
| PaddleOCR | Распознавание текста из сканов | `/ocr` |

**Типичные баги:** 🐛 галлюцинации, OOM на GPU, обрезание ответа, непоследовательные эмбеддинги

### Knowledge Base (RAG)
**Query Plane:** REST API → Retrieval Adapter → RBAC/ABAC фильтр → Weaviate
**Ingestion Plane:** Ingestion API → Redis/RQ очередь → Workers (extract → OCR → chunk → embed → store)

**Типичные баги:** 🐛 нерелевантные документы, ABAC не применяется, ingestion зависает, дубликаты при повторной загрузке

### Безопасность
| Компонент | Назначение |
|-----------|-----------|
| Active Directory | Аутентификация |
| RBAC | Контроль по ролям |
| ABAC | Контроль по атрибутам документов |

**Типичные баги:** 🐛 RBAC фильтр применён к retrieval, но не к LLM-промпту (утечка контекста)

### Наблюдаемость
| Инструмент | QA использует для |
|-----------|------------------|
| Langfuse | Отладка AI-поведения, проверка промптов, просмотр трейсов |
| Prometheus | Мониторинг латентности, ошибок, ресурсов |
| Grafana | Визуальный мониторинг при тестировании |

## 1.4 Основы AI для QA

### LLM (Large Language Model)
Нейронная сеть, обученная на огромных объёмах текста. Предсказывает следующее слово.
- **Вероятностная** — один и тот же вопрос может дать разные ответы
- **Галлюцинирует** — генерирует уверенный, но фактически неверный контент
- Наша платформа использует **Llama 3** (open-source) через vLLM

### RAG (Retrieval-Augmented Generation)
Без RAG: LLM отвечает из «памяти» → галлюцинация
С RAG: сначала находим документы, затем LLM отвечает на их основе → заземлённый ответ

### Эмбеддинги
Числовые представления (векторы) смысла текста. Похожие тексты → близкие векторы.

### Агент
AI-система, которая: получает задачу → планирует → использует инструменты → собирает ответ.

### MCP
Стандартизированный протокол связи агент-инструмент. Аналогия: USB для AI-инструментов.

## 1.6 Дерево отладки

```
Пользователь сообщает о неверном ответе
    │
    ├── Пустой ответ/ошибка? → Логи Backend → Agent → MCP Server
    ├── Галлюцинация? → Langfuse трейс → Правильные ли документы найдены?
    │   ├── Нет документов → Проблема KB (эмбеддинги? индекс?)
    │   ├── Не те документы → Проблема качества retrieval
    │   └── Правильные документы, но неверный ответ → Проблема LLM
    ├── Утечка данных? → RBAC/ABAC фильтр в KB
    └── Медленный ответ? → Grafana → какой компонент тормозит?
```

## 1.7 Ключевые термины

| Термин | Определение | Зачем QA |
|--------|------------|----------|
| LLM | Большая языковая модель | Генерирует ответы |
| RAG | Retrieval-Augmented Generation | Предотвращает галлюцинации |
| Эмбеддинг | Числовой вектор смысла текста | Обеспечивает семантический поиск |
| Weaviate | Векторная БД | Хранит чанки документов |
| Агент | AI-система с планированием и инструментами | Ядро оркестрации |
| MCP | Протокол агент-инструмент | Как агент вызывает инструменты |
| Галлюцинация | Правдоподобный, но фактически неверный контент | Главная категория AI-багов |
| Чанкинг | Разбиение документов на части | Влияет на качество поиска |
| Ingestion | Процесс добавления документов в KB | Pipeline: extract → chunk → embed → store |
| Langfuse | Платформа наблюдаемости для LLM | Трейсы, промпты, оценки |
| RBAC/ABAC | Контроль доступа по ролям и атрибутам | Кто какие документы видит |
